1. Routing — Comment organiser les pages ?

Chaque statut a sa propre URL. Mais comment structurer les routes dans Next.js ?

A — Route groups
Les statuts freelance partagent un layout (header, sidebar). Les salariés ont le leur.
app/ ├── (freelance)/ │ ├── layout.tsx ← header + sidebar freelance │ ├── micro/page.tsx → /micro │ ├── ei/page.tsx → /ei │ ├── sasu/page.tsx → /sasu │ ├── holding/page.tsx → /holding │ └── comparer/page.tsx ├── (salarie)/ │ ├── layout.tsx ← header + sidebar salarié │ ├── salarie/page.tsx → /salarie │ └── portage/page.tsx → /portage └── page.tsx → / (landing)
  • Layout partagé par groupe (DRY)
  • (freelance) invisible dans l'URL
  • Chaque groupe a son propre header/sidebar
B — Routes plates
Toutes les routes au même niveau, layout unique global.
app/ ├── micro/page.tsx → /micro ├── ei/page.tsx → /ei ├── sasu/page.tsx → /sasu ├── holding/page.tsx → /holding ├── salarie/page.tsx → /salarie ├── portage/page.tsx → /portage ├── comparer/page.tsx → /comparer ├── layout.tsx ← un seul layout └── page.tsx → / (landing)
  • Plus simple (pas de route groups)
  • Moins de fichiers layout
  • Pas de layout spécifique par type
  • Header/sidebar identique CDI et freelance

2. Navigation entre statuts — Tabs ou Links ?

Quand l'utilisateur veut passer de /micro à /sasu, comment ça marche visuellement ?

A — Pills-as-Links + transition
Les pills sont des <Link> Next.js. Navigation client-side rapide (~200ms). Skeleton animation pendant le chargement.
Micro EI EURL → clique → SASU
simulateur-freelance.vercel.app/sasu?ca=100000&parts=2
Navigation instantanée, URL change, skeleton 200ms puis cascade
  • URL propre par statut (SEO)
  • Back button fonctionne
  • Skeleton comble le gap visuel
  • Pas de morph fluide entre statuts
B — Tabs-as-State (pas de routing)
Les pills changent l'état local. Morph fluide de la cascade. Mais l'URL ne change pas de route.
Micro EI EURL SASU ←
simulateur-freelance.vercel.app/freelance?statut=sasu&ca=100000
Morph instantané, cascade s'anime en place, 0ms
  • Morph fluide cascade (wow)
  • 0ms de transition
  • URL moins propre (?statut=sasu)
  • SEO : une seule page pour 5 statuts
  • Pas de back button entre statuts

4. Transition entre pages — Quel effet ?

Quand l'utilisateur navigue de /micro à /sasu, que voit-il ?

A — Fade + stagger cascade
La zone cascade fait un fade rapide (100ms), puis les nouvelles cartes apparaissent en stagger (80ms chaque). Header/sidebar restent fixes.
/micro fade out → /sasu cascade stagger in
  • Sensation de continuité
  • Stagger = "les données arrivent"
  • Header stable = pas de saut
B — Cut direct (pas de transition)
Le contenu change instantanément. Pas d'animation entre pages. Simple et rapide.
Tout apparaît d'un coup, 0ms
  • Le plus rapide possible
  • Zéro complexité d'animation
  • Sensation de "saut" entre pages
  • Moins premium

5. Flow pédagogique — Quand l'afficher ?

Le schéma animé @xyflow (le "wow") apparaît quand exactement ?

A — Bouton contextuel dans la cascade
Un bouton discret "Comprendre la mécanique" apparaît sous la cascade, uniquement pour les statuts complexes (SASU, Holding, EURL+IS).
Après la cascade NET...
🔍 Comprendre la mécanique
Voir comment l'argent circule dans la structure
  • Pas intrusif — l'utilisateur choisit
  • Lazy-load (0KB initial)
  • Contextuel : pas visible pour Micro
B — Onglet permanent "Architecture"
Un onglet fixe à côté de "Cascade" permet de basculer à tout moment.
Cascade Architecture
Toujours visible, même pour Micro (même si le flow est simple)
  • Toujours accessible
  • Découvrable facilement
  • Inutile pour Micro (flow trivial)
  • Charge cognitive supplémentaire
